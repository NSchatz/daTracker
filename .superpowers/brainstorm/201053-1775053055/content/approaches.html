Architecture Approaches

Three options for the backend stack powering your location tracker

A

Go + PostgreSQL/PostGIS + WebSockets

RECOMMENDED

Lightweight Go backend with PostGIS for geospatial queries, WebSockets for real-time location push, and FCM for background notifications.

Pros

  • Go compiles to a single binary — tiny Docker image, runs on a Pi
  • PostGIS handles geofence math (point-in-polygon) natively
  • WebSockets give instant map updates with minimal overhead
  • Low memory footprint (~20-50MB for this scale)
  • Excellent concurrency model for real-time connections

Cons

  • More verbose than Python/Node for CRUD operations
  • PostGIS adds complexity vs plain Postgres
B

Node.js (Fastify) + PostgreSQL/PostGIS + Socket.io

JavaScript/TypeScript backend with the same PostGIS foundation, using Socket.io for real-time.

Pros

  • Large ecosystem, easy to find libraries
  • Socket.io handles reconnection/fallback automatically
  • TypeScript provides good type safety

Cons

  • Higher memory baseline (~80-150MB with Node)
  • node_modules adds Docker image bloat
  • Single-threaded — less natural for concurrent connections
C

Rust (Axum) + PostgreSQL/PostGIS + WebSockets

Maximum performance and safety with Rust. Same PostGIS + WebSocket architecture as Option A.

Pros

  • Lowest resource usage of all options
  • Memory safety guarantees
  • Excellent WebSocket performance

Cons

  • Significantly slower development speed
  • Steeper learning curve, harder to maintain
  • Compile times slow down iteration
  • Overkill for 2-10 users

Why I recommend Go (Option A)

For a self-hosted location tracker serving 2-10 users, Go hits the sweet spot: it's resource-efficient enough to run on minimal hardware, has excellent stdlib support for HTTP/WebSockets, and the single-binary deployment makes Docker images tiny. PostGIS is the common thread across all options — it's the right tool for geospatial work regardless of language. The verbosity tradeoff is minor given the app's scope.

Shared across all approaches

  • Database: PostgreSQL + PostGIS (geofence calculations, location history with spatial indexing)
  • Push notifications: Firebase Cloud Messaging (FCM) for geofence alerts when app is backgrounded
  • Maps: OpenStreetMap tiles (free, self-hostable) via the Android app's map SDK
  • Auth: Simple JWT-based with invite tokens for new members
  • Deployment: Docker Compose (app server + PostgreSQL/PostGIS)
  • Mobile: Native Kotlin with Google's Fused Location Provider for adaptive tracking
